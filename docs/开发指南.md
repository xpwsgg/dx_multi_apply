# 开发指南

## 1. 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 框架 | Tauri 2 | 跨平台桌面应用框架 |
| 前端 | React 19 + TypeScript | 单页应用 |
| 构建 | Vite 7 | 前端构建工具 |
| 后端 | Rust | Tauri 后端逻辑 |
| HTTP | reqwest | Rust HTTP 客户端 |
| 数据库 | rusqlite (bundled) | 本地 SQLite 存储提交记录 |
| 包管理 | pnpm 9 | 前端依赖管理 |

## 3. 环境搭建

### 前置条件

- [Node.js](https://nodejs.org/) 22+
- [pnpm](https://pnpm.io/) 9+
- [Rust](https://www.rust-lang.org/tools/install) (stable)
- 系统依赖参考 [Tauri 官方文档](https://v2.tauri.app/start/prerequisites/)

### 安装依赖

```bash
pnpm install
```

### 开发模式

```bash
pnpm tauri dev
```

启动后会同时运行 Vite 开发服务器（热更新）和 Tauri 原生窗口。

### 生产构建

```bash
pnpm tauri build
```

构建产物：
- macOS: `src-tauri/target/release/bundle/dmg/` 下的 `.dmg` 文件
- Windows: `src-tauri/target/release/` 下的可执行文件（免安装版）以及 `bundle/` 下的安装包

## 4. 核心架构

### 4.1 前后端通信

前端通过 Tauri 的 `invoke` 调用后端 Rust 命令，后端通过 `emit` 向前端发送事件。

```
前端 (React)                     后端 (Rust)
    │                                │
    ├── invoke("start_batch_submit") ──→ start_batch_submit()
    │                                │     ├── 遍历日期
    │                                │     ├── 调用 submit API
    ←── emit("batch-log", payload) ──┤     └── 发送日志事件
    │                                │
    ├── invoke("query_visitor_status") ─→ query_visitor_status()
    ←── 返回 Vec<VisitorStatusRecord> ─┤
```

### 4.2 Tauri 命令清单

| 命令 | 功能 | 文件 |
|------|------|------|
| `fetch_visitor_info` | 查询访客信息 | lib.rs |
| `fetch_reception_info` | 查询接待人信息 | lib.rs |
| `start_batch_submit` | 启动批量提交 | lib.rs |
| `stop_batch_submit` | 停止批量提交 | lib.rs |
| `get_existing_dates` | 查询已提交日期 | lib.rs |
| `login_with_code` | 短信验证码登录 | lib.rs |
| `check_token` | 验证 token 有效性 | lib.rs |
| `get_token_status` | 获取当前 token 状态 | lib.rs |
| `clear_token` | 清除登录状态 | lib.rs |
| `query_visitor_status` | 查询预约记录 | lib.rs |
| `load_form_state` | 加载保存的表单状态 | lib.rs |

### 4.3 关键 API 接口

所有 API 请求发往 `https://dingtalk.avaryholding.com:8443/dingplus/visitorConnector/`

| 接口路径 | 功能 | 认证 |
|----------|------|------|
| `sendCode` | 发送验证码 | 无 |
| `visitorLogin` | 登录 | 无 |
| `visitorStatus` | 查询预约记录 | 需要 acToken |
| 宜搭表单 API | 查询访客/接待人信息、提交申请单 | 无 |

### 4.4 数据存储

| 存储 | 位置 | 内容 |
|------|------|------|
| SQLite | `$APP_DATA/records.db` | 已提交记录（用于跳过重复日期） |
| JSON 文件 | `$APP_DATA/token.json` | acToken + 手机号 + 获取时间 |
| JSON 文件 | `$APP_DATA/form_state.json` | 上次填写的表单内容 |
| JSONL 文件 | `$APP_DATA/batch.log` | 操作日志 |

## 5. CI/CD

### 发布流程

1. 更新版本号（需同步三处）：
   - `package.json` → `version`
   - `src-tauri/tauri.conf.json` → `version`
   - `src-tauri/Cargo.toml` → `version`

2. 提交代码并创建 tag：
   ```bash
   git add .
   git commit -m "chore(release): bump version to X.Y.Z"
   git tag vX.Y.Z
   git push origin main --tags
   ```

3. GitHub Actions 自动构建并创建 Release，产出物包括：
   - macOS `.dmg` (aarch64 + x64)
   - Windows `.msi` + `.exe` 安装包
   - Windows `BatchApply-portable-x64.exe` 免安装版

### Workflow 触发条件

CI 构建配置在 `.github/workflows/release.yml`，仅以下方式触发：

1. **推送 tag（自动触发）**：推送 `v*` 格式的 tag 会自动触发构建和发布
   ```bash
   # 完整流程：更新版本号 → 提交 → 打 tag → 推送
   git add package.json src-tauri/Cargo.toml src-tauri/tauri.conf.json
   git commit -m "chore(release): bump version to X.Y.Z"
   git tag vX.Y.Z
   git push origin main vX.Y.Z
   ```

2. **手动触发（workflow_dispatch）**：在 GitHub Actions 页面点击 "Run workflow" 按钮手动触发，或通过 CLI：
   ```bash
   gh workflow run release.yml
   ```

> **注意**：直接 `git push` 代码到 `main` 分支不会触发构建，必须推送 `v*` tag 或手动触发。

## 6. 常见问题

### 登录相关

- **验证码获取失败**：检查手机号是否正确，确认网络可达 `dingtalk.avaryholding.com`
- **登录失效**：acToken 有时效，失效后需重新登录
- **token 验证逻辑**：仅当 API 返回 `code=200` 时视为有效，其他情况一律视为失效

### 提交相关

- **跳过已存在日期**：后端根据 SQLite 记录自动跳过同一接待人 + 同一日期的重复提交
- **随机等待**：每次提交间隔 3~8 秒随机延迟，防止触发接口限流
- **并发保护**：后端有原子锁防止重复提交，前端按钮在任务期间禁用

### 构建相关

- **Windows 免安装版**：构建后从 `target/release/` 获取 exe，前端资源已嵌入
- **WebView2 依赖**：Windows 10 1803+ / Windows 11 已自带，无需额外安装
