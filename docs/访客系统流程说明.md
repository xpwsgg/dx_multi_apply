# 访客系统流程说明

## 1. 文档目的

本文基于 `/Users/xiao/Desktop/访客系统` 下提供的资料整理，说明当前访客系统的核心业务流程、关键接口和本工具中对应的使用方式，便于后续排查登录、提交流程和预约记录查询问题。

## 2. 核心业务流程

### 2.1 登录取 token

1. 打开移动端登录页：

```text
https://iw68lh.aliwork.com/o/fk_login_app
```

2. 使用申请人手机号请求验证码。
3. 将手机号和验证码提交到 `visitorLogin` 接口执行登录。
4. 登录成功后跳转到首页：

```text
https://iw68lh.aliwork.com/o/fkxt_index_app?account=<申请人手机号>
```

5. 首页在已登录状态下会把 `acToken` 写入浏览器 `localStorage`。
6. 后续“查询预约记录”等接口需要携带 `acToken`。

### 2.2 填写访客申请单

1. 申请人手机号作为 `account` 参数参与后续请求。
2. 填写访客信息、接待人信息和访问日期。
3. 批量提交工具按日期串行提交申请单。

### 2.3 查询预约记录

1. 先保证当前手机号对应的登录态有效。
2. 调用 `visitorStatus` 接口。
3. 请求体中必须带：
   - `visitorIdNo`: 访客身份证号
   - `regPerson`: 申请人手机号
   - `acToken`: 登录成功后获得的 token
4. 返回 `code = 200` 表示登录有效且查询成功。
5. 返回 `code = 401` 表示未登录或登录失效，需要重新登录。

## 3. 关键接口说明

### 3.1 请求验证码

请求地址：

```text
POST https://dingtalk.avaryholding.com:8443/dingplus/visitorConnector/sendCode
```

请求体示例：

```json
{
  "phoneNum": "17849759601",
  "areaCode": "86",
  "region": ""
}
```

响应示例：

```json
{
  "code": 200,
  "message": "SUCCESS",
  "data": {
    "code": 875705
  }
}
```

说明：

- 当前系统返回的 `data.code` 就是后续登录时要提交的验证码。

### 3.2 登录

请求地址：

```text
POST https://dingtalk.avaryholding.com:8443/dingplus/visitorConnector/visitorLogin
```

请求体示例：

```json
{
  "phoneNum": "17849759601",
  "code": "875705"
}
```

说明：

- 单独调用接口只能说明服务端接受了登录请求。
- 真正用于后续查询的 `acToken` 不是这个接口直接返回的字段，而是登录完成后访问首页时写入浏览器 `localStorage`。

### 3.3 查询预约记录

请求地址：

```text
POST https://dingtalk.avaryholding.com:8443/dingplus/visitorConnector/visitorStatus
```

请求体示例：

```json
{
  "visitorIdNo": "320882198910054470",
  "regPerson": "17849759601",
  "acToken": "4D682F3708A4636794F0B0F06A8D1475CDD3E78C4D7682FA514E43CE2F01FC49"
}
```

成功响应特征：

- `code = 200`
- `data` 为预约记录数组

失效响应特征：

```json
{
  "code": 401,
  "message": "FAIL",
  "data": "FAIL"
}
```

## 4. 本工具中的对应关系

### 4.1 手机号

- 对应界面中的“申请人手机号”。
- 对应原系统资料中的 `account` / `regPerson`。

### 4.2 登录态

- 本工具需要先完成登录，拿到 `acToken`。
- `acToken` 会保存在本地文件中，供“查询预约记录”功能复用。

### 4.3 查询预约记录

- 界面中点击查询时，会使用本地已保存的 `acToken` 调用 `visitorStatus`。
- 如果返回 401，则应提示重新登录。

## 5. 关键注意事项

1. `visitorLogin` 成功不等于已经拿到 `acToken`。
2. `acToken` 的来源是“登录成功后的首页浏览器存储”，不是 Rust 侧 HTTP 响应体。
3. `account`、`regPerson`、申请人手机号在当前业务里是同一语义。
4. 如果访问首页时没有建立正确登录态，页面就不会写入 `acToken`，随后查询预约记录必然失败。

## 6. 排查建议

当出现“未登录”“登录失效”“acToken 为空”时，优先按下面顺序检查：

1. 是否成功请求到验证码。
2. 是否成功执行 `visitorLogin`。
3. 登录后的首页是否在同一个浏览器会话中打开。
4. 首页是否成功写入 `localStorage.acToken`。
5. `visitorStatus` 请求中的 `regPerson` 和 `acToken` 是否匹配。
