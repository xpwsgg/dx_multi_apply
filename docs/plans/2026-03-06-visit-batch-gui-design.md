# 2026-03-06 到访批量提交 GUI 设计

## 1. 背景与目标

现有项目已经具备 Rust/Tauri 侧批量提交流程与日期 payload 替换能力，但前端仍是默认模板页面，无法直接用于业务操作。目标是提供一个简单 GUI，让用户通过“开始日期 + 结束日期”生成日期区间，允许手动删除个别日期后发起批量提交。

核心目标：
- 支持闭区间日期展开（包含开始日和结束日）。
- 仅修改既有请求 payload 中“到访日期”“到访日期文本”两个值。
- 保持现有后端批量提交策略不变（升序、随机间隔、失败即停）。
- 日志中完整展示服务端响应数据，便于调试。

## 2. 约束与已有能力

### 2.1 现有后端能力（保持）
- `start_batch_submit(dates: Vec<String>)`：按日期排序后逐个提交（`src-tauri/src/lib.rs`）。
- 成功后随机等待 `120..=180` 秒；失败立即返回并中止。
- `stop_batch_submit()`：支持手动停止。
- `build_payload_for_date(date)`：仅替换 payload 中：
  - `label == "到访日期"` 的 `DateField.fieldData.value`
  - `label == "到访日期文本"` 的 `TextField.fieldData.value`

### 2.2 新增成功判定规则
现有提交逻辑仅依据 HTTP 2xx 判定成功，需补充为业务成功判定：
1. HTTP 状态码为 2xx；
2. 响应体可解析为 JSON；
3. JSON 中 `success === true`。

否则视为失败并停止后续提交。

## 3. 方案对比与结论

### 方案 A（采用）前端展开区间 + 可编辑列表
- 前端根据起止日期生成闭区间日期数组；用户可删除个别日期。
- 最终把数组传给既有 `start_batch_submit`。

优点：改动最小、交互直观、实现快、风险低。
缺点：区间生成逻辑在前端。

### 方案 B 后端展开区间
- 新增后端命令接收起止日期并返回列表，再由前端编辑并提交。

优点：规则集中于后端。
缺点：往返与接口复杂度更高，超出“简单 GUI”目标。

### 方案 C 前后端双重展开
- 前端展示 + 后端重复展开校验。

优点：冗余高。
缺点：重复逻辑与维护成本高，不必要。

**结论：采用方案 A。**

## 4. 交互与信息架构

## 4.1 页面结构
前端 `src/App.tsx` 替换为业务页面，包含：
1. 起止日期输入区
   - 开始日期（date input）
   - 结束日期（date input）
   - 生成日期按钮
2. 待提交日期列表区
   - 逐行显示 `YYYY-MM-DD`
   - 每行“删除”按钮
3. 批量执行区
   - 开始提交按钮
   - 停止提交按钮
4. 日志区
   - 实时显示每次提交结果和服务端完整响应

## 4.2 关键交互
- 用户选择开始/结束日期，点击“生成日期列表”。
- 前端按闭区间展开日期（含首尾），默认升序展示。
- 用户可删除不想提交的日期。
- 点击“开始提交”后进入运行态：
  - 禁止编辑日期输入与列表（避免状态漂移）
  - 可点击“停止提交”
- 后端通过 `batch-log` 推送日志，前端实时渲染。

## 4.3 前端状态模型
- `startDate: string`
- `endDate: string`
- `dates: string[]`（最终提交列表，可删除）
- `isRunning: boolean`
- `logs: Array<{ date?: string; result: string; reason?: string; waitSeconds?: number; responseRaw?: string }>`
- `errorMessage?: string`

## 5. 数据流设计

1. **生成阶段**
   - 输入 `startDate/endDate`
   - 前端校验：非空、`startDate <= endDate`
   - 生成闭区间日期数组写入 `dates`

2. **编辑阶段**
   - 用户删除单个日期
   - `dates` 即时更新
   - 若 `dates` 为空，禁止提交

3. **执行阶段**
   - `invoke("start_batch_submit", { dates })`
   - 后端按既有策略逐日提交
   - 后端 emit `batch-log`，前端追加日志
   - 失败/停止后退出运行态

## 6. 错误处理与可观测性

### 6.1 前端输入错误
- 未选择开始或结束日期：提示“请先选择开始和结束日期”。
- 开始晚于结束：提示“开始日期不能晚于结束日期”。
- 删除后无日期：提示“请至少保留一个日期”。

### 6.2 后端提交失败条件
以下任一成立都为失败：
- HTTP 非 2xx；
- 响应 JSON 解析失败；
- 响应 JSON 中 `success !== true`。

失败时：
- 返回错误信息（含必要上下文）；
- 发出失败日志；
- 立即停止后续日期提交。

### 6.3 日志要求
- 记录每个日期的结果（success / failed / stopped）。
- **完整保留服务端原始响应字符串**（如 `responseRaw`）用于 debug。
- 成功日志可附带 `waitSeconds`。

## 7. 测试与验收标准

### 7.1 前端验收
- 区间 `2026-03-01 ~ 2026-03-03` 展开为 3 条（含首尾）。
- 删除任意日期后列表即时变化。
- 列表为空时不可开始提交。
- 运行态按钮状态正确（开始禁用、停止可用）。
- 日志按事件顺序展示。

### 7.2 后端验收
- 维持现有顺序、随机间隔、失败即停行为。
- 业务成功判定新增 `success===true`。
- 响应原文可透传到日志。

### 7.3 手工回归
- 正常成功返回样例中 `success: true` 时判定成功。
- `success: false` 或 `errorMsg` 非空时判定失败并停止。
- 点击停止后应收到 stopped 结果并终止后续提交。

## 8. 非目标
- 不引入复杂日历多选组件。
- 不新增并发提交（仍串行）。
- 不修改 payload 中除“到访日期/到访日期文本”外的字段。

## 9. 实施边界
- 前端主要改动：`src/App.tsx` 与样式文件。
- 后端主要改动：`src-tauri/src/submit_client.rs`（成功判定与响应透传），必要时微调 `src-tauri/src/lib.rs` 的日志 payload 字段。
- 不改变命令名和主流程接口，降低前后端联调成本。
